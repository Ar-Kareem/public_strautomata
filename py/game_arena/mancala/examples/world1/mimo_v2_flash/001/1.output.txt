Here is a summary of the reasoning process to design the Kalah policy:

1.  **Understand the Goal**: The objective is to implement a function `policy(you, opponent)` that selects a legal move (0-5) to maximize winning chances in a 6-house Kalah game.

2.  **Analyze the Game Mechanics**:
    *   **Sowing**: Seeds are distributed counter-clockwise, skipping the opponent's store.
    *   **Extra Move**: Landing in your own store grants an extra turn.
    *   **Capture**: Landing in an empty house on your side allows capturing that seed and seeds from the opposite opponent house, adding them to your store.
    *   **Winning**: The game ends when one side is empty. Remaining seeds on the opponent's side go to the opponent's store.

3.  **Strategy Selection**:
    *   *Greedy*: Immediate captures and extra moves are highly beneficial.
    *   *Blocking*: Preventing the opponent from capturing is crucial.
    *   *Lookahead*: Since the branching factor is moderate (usually <6) and the depth is limited by seed count, a Minimax search with Alpha-Beta pruning is feasible within the 1-second time limit.

4.  **Implementation Details**:
    *   **State Representation**: The board is effectively 14 houses (0-6 for me, 7-13 for opponent) plus 2 stores (14 for me, 15 for opponent), or simply keeping the two lists provided but adapting them for the simulation engine.
    *   **Simulation**: Create a function to apply a move to a copy of the state. It must handle sowing, extra moves, and captures recursively or iteratively.
    *   **Evaluation Function**: A simple heuristic is sufficient for a shallow search: `(MyStore - OpponentStore)`.
    *   **Search**: Minimax with Alpha-Beta pruning to a specified depth. The depth should be dynamic based on the number of seeds remaining to fit the time limit.

5.  **Code Structure**:
    *   Define the `policy` function.
    *   Define helper functions for `simulate_move` and `minimax`.
    *   Inside `policy`: determine search depth, run the search on available moves, and return the best one. If no time for deep search or simple heuristic is needed, fall back to a greedy approach (prefer captures -> extra moves -> higher sowing count).

6.  **Refinement**: Ensure all edge cases (like ending the game and capturing) are handled correctly in the simulation logic.

        return sum(board[:6]) == 0 or sum(board[7:13]) == 0

        my_store = board[6]
        opp_store = board[13]
        # Remaining seeds on the empty side go to the owner of that side
        if sum(board[:6]) == 0:
            opp_store += sum(board[7:13])
        else:
            my_store += sum(board[:6])
        return my_store - opp_store

        WAIT: The input is 'you' and 'opponent' lists.

        Current Board State for Search:

        else:

        maximizingPlayer: True if we are evaluating for the player whose turn it is in 'board'

            return score, None

                    eval_score, _ = minimax(new_board, depth - 1, alpha, beta, False)

                if new_board[13] > board[13]: # Landed in store
