The strategy for this policy is to implement a sophisticated AI for Havannah that combines aggressive threat detection with defensive blocking and winning pattern recognition. Given the complexity of the three winning conditions (ring, bridge, fork), we prioritize moves that:

1. **Immediately win** the game if a winning structure can be completed in one move.
2. **Block opponent’s immediate threats** — if the opponent has a near-win (e.g., 4 stones in a potential bridge or fork), we block.
3. **Extend strong positional lines** — prioritize moves that extend our own connected groups toward edges or corners for bridge/fork potential.
4. **Create multiple threats (forking)** — play moves that simultaneously threaten two different winning conditions, forcing the opponent into an impossible defense.
5. **Use symmetry and central control** — prioritize central and near-central moves early to maximize influence and connection potential, as the central area has the most hexagonal neighbors.
6. **Detect ring potential** — track loops formed by our stones and complete a ring if possible.
7. **Use breadth-first search (BFS) and depth-limited A* for connectivity analysis** — to compute connected components, edge access, and potential ring closures efficiently.

Given the 15x15 grid and hexagonal topology, we define the 6 neighbor directions using (dr, dc) offsets that map the 2D grid to hexagonal adjacency. For a cell (r, c), its neighbors are:
- (r-1, c-1) — NW
- (r-1, c)   — NE
- (r, c-1)   — W
- (r, c+1)   — E
- (r+1, c)   — SW
- (r+1, c+1) — SE

We use this to compute component connectivity (for bridges and forks) and ring detection via cycle detection in graph traversal.

We limit search depth to 2-ply for speed within 1s, using heuristic evaluation (number of connections to edges, centrality, threat counts). The evaluation function weights:
- Ring potential: +10 per stone in partial ring
- Bridge: +8 per stone connected to corner + 20 if two corners connected
- Fork: +6 per edge connection + 30 if three edges connected
- Centrality: +1 per move closer to center (to favor center control)

We use a transposition table for caching board states and prioritize moves with highest heuristic gain, falling back to random valid move only if no beneficial moves are found (should be rare).

def are_connected(stone_set, start, end):
    """Check if two stones are connected via a path of stones in stone_set using BFS."""
    if start == end:
        return True
    if start not in stone_set or end not in stone_set:
        return False
    queue = deque([start])
    visited = {start}
    
    while queue:
        r, c = queue.popleft()
        if (r, c) == end:
            return True
        for nr, nc in get_neighbors(r, c):
            if (nr, nc) in stone_set and (nr, nc) not in visited:
                visited.add((nr, nc))
                queue.append((nr, nc))
    return False

def policy(me: list[tuple[int, int]], opp: list[tuple[int, int]], valid_mask) -> tuple[int, int]:

    me_set = set(me)
    opp_set = set(opp)
    
    # 1. Check for immediate win: can we complete a ring, bridge, or fork in one move?
    
    # Check if any move completes a bridge
    bridge_win_moves = is_bridge_possible(me, valid_mask)
    if bridge_win_moves:
        return bridge_win_moves[0]  # Return first winning move
    
    # Check if any move completes a fork
    fork_win_moves = find_fork_threats(me, valid_mask)
    # Fork win: if placing a stone creates 3 edge connections
    for move in fork_win_moves:
        if not (move in me_set or move in opp_set) and valid_mask[move[0]][move[1]]:
            # Check if this move makes 3 edges
            me_temp = me_set | {move}
            components = find_connected_components(list(me_temp), valid_mask)
            for comp in components:
                if move in comp:
                    edges = get_edge_set(comp)
                    if len(edges) >= 3:
                        return move
    
    # Check for ring wins (simplified — assume we win if we complete a loop)
    ring_win_moves = find_ring_threats(me, valid_mask)
    if ring_win_moves:
        return ring_win_moves[0]
    
    # 2. Block opponent's immediate wins (if they are about to win)
    opp_bridge_moves = is_bridge_possible(opp, valid_mask)
    if opp_bridge_moves:
        return opp_bridge_moves[0]  # Block the bridge
    
    opp_fork_moves = find_fork_threats(opp, valid_mask)
    for move in opp_fork_moves:
        if not (move in me_set or move in opp_set) and valid_mask[move[0]][move[1]]:
            # See if this move creates fork for opp
            opp_temp = opp_set | {move}
            components = find_connected_components(list(opp_temp), valid_mask)
            for comp in components:
                if move in comp:
                    edges = get_edge_set(comp)
                    if len(edges) >= 3:
                        return move
    
    # 3. Look for beneficial moves: create threats, improve position
    candidate_moves = []
    
    # Generate all valid moves
    for r in range(15):
        for c in range(15):
            if valid_mask[r][c] and (r, c) not in me_set and (r, c) not in opp_set:
                # Heuristic evaluation
                score = evaluate_move(me, opp, (r, c), valid_mask)
                candidate_moves.append((score, (r, c)))
    
    # Sort by score descending
    candidate_moves.sort(key=lambda x: x[0], reverse=True)
    
    # Return best move if exists
    if candidate_moves:
        return candidate_moves[0][1]
    
    # Fallback: if no good moves, pick any valid move (shouldn't happen in normal play)
    for r in range(15):
        for c in range(15):
            if valid_mask[r][c] and (r, c) not in me_set and (r, c) not in opp_set:
                return (r, c)
    
    # Last resort — this should never be reached because valid_mask must have at least one
    raise RuntimeError("No valid move available!")
